Haphazard C++ Style Guide

Motivation

Having a consistent style means:
  • Less mental energy is spent on parsing style differences between files.
  • Code is likely to be easier to understand than it might otherwise be.
  • It’s easier to work on files even when you didn’t initially write them.
The purpose of a style guide is not to impose arbitrary restrictions on code (though some style decisions are arbitrary), but to keep things as clear and consistent as possible.

Whenever there’s any confusion or doubt about what exactly to do, ask yourself what another programmer coming to look at your file would need to make everything clear, understandable, and as hassle-free for them as possible.

Spacing

Indentation should be done with tabs, but any spacing once at the proper indentation level should use spaces: 

int main()
{
	int variables = 0;
	int      that = 1;
	int      need = 2;
	int    lining = 3;
	int        up = 4;
	
	if (that != need)
	{
		int moreIndentedCode = 5;
	}
	
	return 0;
}
Binary operators should always have spaces around them: 

// Good usage.
int good = (1 + 2) * 14;

// Bad usage
int bad=(1+2)*14;
Unary operators should never have spaces after them: 

// Good usage.
int good = -5;

// Bad usage
int bad = - 5;

Naming

Names should strike a balance between not taking up too much space and clearly indicating what that name represents, with an emphasis towards the latter. The more localized to a scope a name is, the shorter it can be. Variables should be written in lowerCamelCase: 

// This is correct:
int someGoodVariable;

// These should be avoided:
int SomeBadVariable;
int another_bad_variable;
Function names should also be written in lowerCamelCase: 

// Correct.
float secondsToHours(int time);

// Incorrect
float SecondsToHours(int seconds);
float seconds_to_hours(int seconds);
Constants should be in ALL_CAPS: 

// Correct.
const int GOOD_CONSTANT = 0;

// Incorrect
const int badConstant = 1;
const int ANOTHERBADCONSTANT = 2;
Private member variables should begin with "m_": 

class ExampleClass
{
public:
    bool publicVariable;

private:
    int m_somePrivateMemberVariable;
    float m_shorterName;
}

Comments

Comments are (generally) good.

When to Comment

In general, there are two places where comments should be used:
  • Comments can serve as “landmarks” of sorts in the code, telling the reader what the next handful of lines will be doing: 

// Check each enemy and delete it if it's dead.
for (int i = 0; i < objectCount; ++i)
{
  if (objectArr[i]->health <= 0)
  {
    delete objectArr[i];
    objectArr[i] = nullptr;
  }
}
  • Comments can explain why something is being done the way it is. 
                    
// Sprite sizes not divisible by 4 mess up the lighting buffer.
// Return early if we run into any.
if (sprite->width % 4 != 0 || sprite->height % 4 != 0)
{
  return;
}
Obviously, put comments wherever you think they’re necessary, but the majority of comments should fit into one of the categories above.

When NOT to Comment

Comments where there shouldn’t be any just clutter things up. Let the code speak for itself where it can.

General Formatting

Writing in C++ means there are a lot of little style decisions that need to be made. These are less philosophical than, say, commenting, but they need to be said. Here’s a bunch of those.

Brackets

Brackets always go on their own line: 

// Do this
if (conditional)
{
  // CODE
}
 
// Not this
if (conditional){
  // CODE
}
This is intended to line up the brackets so they can be more easily matched with one another.

File Header Comments

Every file should have a header comment. Like so: 
   
/*******************************************************
FILE: .cpp
 
PRIMARY AUTHOR: 
 
PROJECT: 
 
Copyright 2017 DigiPen (USA) Corporation.
*******************************************************/
Personally I think file header comments aren’t very good, but DigiPen requires this info so we get to put it there.

Function Header Comments

Function header comments are not required, but you are highly encouraged to put at least a short comment above each function describing its purpose.

Also remember to use comments to indicate usage, return values, etc. when they're not immediately obvious.
